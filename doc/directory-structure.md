OpenSSL Repository Structure
============================

This document provides an overview of the OpenSSL repository's
directory structure, explaining what the major directories and
files are, their purpose, and how they relate to each other.

The canonical location of this file is `doc/directory-structure.md`
in the OpenSSL source distribution.

Repository Tree Overview
------------------------

Here is a high-level view of the repository structure:

```
openssl/
├── apps/             # Command-line application source
├── Configurations/   # Build platform definitions and templates
├── crypto/           # Cryptographic library (libcrypto)
├── demos/            # Example programs
├── dev/              # Release engineering scripts
├── doc/              # Documentation (man pages, guides, designs)
├── exporters/        # CMake and pkg-config files
├── external/         # External Perl modules
├── fuzz/             # Fuzzing harnesses and corpora
├── include/          # Public and internal header files
├── ms/               # Microsoft Windows specific files
├── os-dep/           # OS-dependent platform abstractions
├── providers/        # Provider implementations (default, FIPS, etc.)
├── ssl/              # SSL/TLS/DTLS/QUIC library (libssl)
├── test/             # Test suite
├── util/             # Build and development utilities
└── VMS/              # OpenVMS specific files
```

Root-Level Files
----------------

* `README.md` - Project overview with links to documentation
* `INSTALL.md` - Detailed build and installation instructions
* `Configure` - The main build configuration script (Perl)
* `config` - A wrapper that detects the platform and calls
  `Configure`
* `LICENSE.txt` - The Apache 2.0 license (OpenSSL 3.0+)
* `CHANGES.md` - Comprehensive changelog
* `NEWS.md` - Curated list of major changes per release
* `VERSION.dat` - The single source of truth for the version number
* `CONTRIBUTING.md` - Guidelines for contributing to OpenSSL
* `HACKING.md` - How to add custom modifications to the source
* `NOTES-*.md` - Platform-specific build notes (UNIX, WINDOWS,
  ANDROID, etc.)
* `README-*.md` - Additional READMEs for specific components
  (FIPS, QUIC, ENGINES, PROVIDERS)
* `configdata.pm.in` - Template used during build configuration

Major Directories
-----------------

### crypto/ - The Cryptographic Library (libcrypto)

The source for `libcrypto`, organized into many subdirectories
by algorithm family or subsystem. Each subdirectory has its own
`build.info` file. Inside you will find subdirectories for
different cryptographic functions as well as other supporting
code. Some notable examples (not an exhaustive list):

* `crypto/aes/` - AES encryption
* `crypto/bn/` - Big number arithmetic
* `crypto/evp/` - The EVP high-level cryptographic API. This is
  the main API through which applications access
  ciphers, digests, signatures, key exchange, KDFs, MACs, and key
  management. It dispatches to provider implementations.
* `crypto/bio/` - The BIO (Basic I/O) abstraction layer. Files
  prefixed `bss_` are source/sink BIOs (file, socket, memory) and
  files prefixed `bf_` are filter BIOs (buffering, base64, logging).
* `crypto/asn1/` - Core ASN.1 DER/BER parsing and encoding
* `crypto/x509/` - X.509 certificate and CRL handling
* `crypto/rsa/`, `crypto/ec/`, `crypto/dh/`, `crypto/dsa/` -
  Algorithm-specific implementations
* `crypto/ml_kem/`, `crypto/ml_dsa/`, `crypto/slh_dsa/`,
  `crypto/lms/` - Post-quantum algorithm implementations
* `crypto/property/` - Property-based algorithm lookup for the
  provider architecture
* `crypto/objects/` - OID definitions and lookup tables (contains
  generated files; run `make update` after editing `objects.txt`)
* `crypto/rand/` - Random number generation (DRBG framework)
* `crypto/err/` - Error recording system (error codes and strings
  are generated by `util/mkerr.pl`)
* `crypto/modes/` - Block cipher mode implementations (GCM, CCM,
  XTS, CTR, OCB, SIV) shared across ciphers

Many C files sit directly in `crypto/` providing core
infrastructure: `context.c` (OSSL_LIB_CTX management),
`provider.c` (provider loading), `core_fetch.c` (algorithm
fetching), `init.c` (library initialization), `mem.c` (memory
allocation), `params.c` (OSSL_PARAM handling), and CPU feature
detection (`cpuid.c`, `armcap.c`, `ppccap.c`).

### ssl/ - The Protocol Library (libssl)

The source for `libssl`, which implements SSL/TLS, DTLS, and
QUIC protocols.

Key files directly in `ssl/` include: `ssl_lib.c` (core connection
lifecycle), `ssl_cert.c` (certificate handling), `ssl_ciph.c`
(cipher suite selection), `ssl_sess.c` (session management),
`s3_lib.c` (TLS cipher suite definitions), `t1_lib.c` (TLS
extensions), `tls13_enc.c` (TLS 1.3 key derivation), `d1_lib.c`
(DTLS core), and `bio_ssl.c` (BIO wrapper for SSL connections).

Notable subdirectories:

* `ssl/statem/` - The TLS/DTLS handshake state machine.
  `statem_clnt.c` and `statem_srvr.c` implement the client and
  server handshake flows. `extensions*.c` files handle TLS
  extension negotiation. This area is actively maintained as new
  protocol features are added.
* `ssl/record/` - The TLS record layer: framing, encryption, and
  MAC operations.
* `ssl/quic/` - The QUIC protocol implementation. This is the
  newest and most actively evolving part of
  libssl. Key files include `quic_impl.c` (main connection logic),
  `quic_channel.c` (connection context), and `quic_txp.c` (packet
  builder). Also includes flow control, stream management,
  congestion control, and QLOG event logging.
* `ssl/rio/` - Reactor I/O: nonblocking event handling and polling
  abstractions used by the QUIC implementation.

### providers/ - Algorithm Providers

OpenSSL 3.0+ uses a provider model where algorithm implementations
are loaded as modules. Five providers are built from this directory:

* **default** (`defltprov.c`) - The standard provider, loaded
  automatically. Contains most algorithm implementations.
* **base** (`baseprov.c`) - Provides encoders, decoders, and store
  loaders. Always loaded alongside other providers.
* **fips** (`fipsprov.c` and `providers/fips/`) - The FIPS 140-3
  validated provider. Built as a separate module with its own
  integrity checks. `self_test.c` and `self_test_kats.c` implement
  the required power-on self-tests (Known Answer Tests).
* **legacy** (`legacyprov.c`) - Deprecated algorithms (MD2, MD4,
  RC2, RC4, Blowfish, CAST, SEED, etc.) not available by default.
* **null** (`nullprov.c`) - A minimal stub provider for testing.

`providers/implementations/` contains all algorithm code, organized
into subdirectories: `ciphers/`, `digests/`,
`signatures/`, `exchange/`, `kem/`, `kdfs/`, `macs/`, `keymgmt/`,
`encode_decode/`, `rands/`, and `storemgmt/`.

`providers/common/` contains utilities shared across providers.

Each provider registers its operations via `OSSL_DISPATCH` arrays
in its query function.

### apps/

Source for the `openssl` command-line tool. Each subcommand has
its own file: `ca.c`, `req.c`, `x509.c`, `dgst.c`, `enc.c`,
`s_client.c`, `s_server.c`, `cms.c`, `pkcs12.c`, `genpkey.c`,
`pkey.c`, `pkeyutl.c`, `fipsinstall.c`, `speed.c`, and many
more. `openssl.c` is the main entry point.

* `apps/lib/` - Shared utility code: `opt.c` (command-line option
  parsing), `apps.c` (common helpers), `s_cb.c` (SSL callbacks for
  s_client/s_server), and platform-specific initialization.
* `apps/include/` - Internal headers for the apps code.

### include/

* `include/openssl/` - Public API headers that get installed with
  `make install`. These define the external API surface of OpenSSL.
  Any new public function must be declared here.
* `include/internal/` - Internal headers shared across libcrypto
  and libssl but not installed. These may change even between
  patch releases and are not part of the public API.
* `include/crypto/` - Internal headers specific to the crypto
  module. Used by code within `crypto/` and `providers/` but not
  part of the public API.

### doc/

* `doc/man1/` - Manual pages for command-line tools
* `doc/man3/` - Manual pages for library functions
* `doc/man5/` - Manual pages for file formats
* `doc/man7/` - Overviews, concepts, and guides.
  Includes the OpenSSL Guide (`ossl-guide-*` pages) and provider
  interface documentation (`provider-*` pages).
* `doc/internal/` - Internal developer documentation. `man3/`
  documents internal APIs; `man7/` has
  architectural overviews.
* `doc/designs/` - Design documents for features including
  post-quantum algorithms, QUIC (`quic-design/` subdirectory),
  EVP extensions, and FIPS support.
* `doc/life-cycles/` - Graphviz diagrams showing object state
  machines for cipher, digest, KDF, MAC, PKEY, and RAND.
* `doc/HOWTO/` - A few legacy how-to documents.

### demos/

Example programs organized by topic: `cipher/`, `digest/`,
`signature/`, `pkey/`, `kdf/`, `mac/`, `encrypt/`, `encode/`,
`bio/`, `cms/`, `smime/`, `sslecho/`, `quic/`, and more. Each
subdirectory has its own `build.info` and can be compiled as part
of the build. `demos/guide/` contains the code samples referenced
by the OpenSSL Guide (`ossl-guide-*` man pages).

### test/

C test source files compiled into test binaries.

* `test/recipes/` - Perl test scripts following the
  naming convention `NN-test_name.t` (e.g., `01-test_sanity.t`).
  These are invoked by `make test` and orchestrate running the
  compiled test binaries.
* `test/testutil/` - A test utility library providing the test
  harness framework: assertions, option parsing, provider loading,
  and a fake random source for deterministic testing.
* `test/ssl-tests/` - TLS/DTLS test configurations generated from
  `.cnf.in` templates.
* `test/certs/` - Test certificates and keys.

Run tests with: `make test TESTS="test_name"`.
See `test/README.md` for more details.

Build System
------------

* `Configure` - The main configuration script (Perl).
* `config` - Platform-detecting wrapper for `Configure`.
* `Configurations/` - Platform definitions and build templates.
  `10-main.conf` defines the base targets (GCC, Clang, MSVC, etc.).
  `50-*.conf` files add platform-specific overrides (Android, iOS,
  MinGW, VMS, etc.). Template files (`unix-Makefile.tmpl`,
  `windows-makefile.tmpl`, `descrip.mms.tmpl`) generate the actual
  build files. `INTERNALS.Configure` documents the system.
* `build.info` files are found throughout the source tree. Each
  directory that contributes source files has one declaring what
  gets built from its contents.
* `exporters/` - Package configuration files for CMake
  (`OpenSSLConfig.cmake`) and pkg-config (`libcrypto.pc`,
  `libssl.pc`, `openssl.pc`).
* `VERSION.dat` - The version number.

Utilities
---------

* `util/find-doc-nits` - Documentation linter. Validates POD
  structure, wording conventions, and cross-references. Run with
  `make doc-nits`.
* `util/mkdef.pl` - Generates linker version scripts and symbol
  export lists from the ordinals files.
* `util/mkerr.pl` - Generates error code infrastructure (reason
  code defines and error string tables).
* `util/libcrypto.num`, `util/libssl.num` - Canonical lists of
  every exported symbol and its ordinal, used to maintain ABI
  compatibility. Run `make update` after adding new public
  functions.
* `util/perl/` - Perl modules used by the build system and tests,
  including `OpenSSL::Test` (test framework) and `TLSProxy/` (a
  TLS protocol implementation in Perl for testing).
* `dev/` - Release engineering: `release.sh` and supporting scripts.

External Modules
----------------

OpenSSL includes several external modules as Git submodules,
mainly used for testing:

* `oqs-provider/` - Post-quantum cryptography provider
* `krb5/` - Kerberos support
* `cloudflare-quiche/` - Cloudflare QUIC implementation (used as
  a reference)

Platform-Specific Code
----------------------

* `os-dep/` - OS abstractions: platform-specific directory access
  (`LPdir_unix.c`, `LPdir_win.c`, etc.) and Apple platform support.
* `ms/` - Microsoft Windows code: the uplink mechanism
  (`applink.c`, `uplink.c`) allowing the OpenSSL DLL to use the
  application's C run time.
* `VMS/` - OpenVMS support scripts and configuration.

Fuzzing
-------

* `fuzz/` - Fuzzing harnesses for ASN.1, X.509, CMS, PKCS12, and
  other parsers, along with seed corpora for each target.

Stability Information
---------------------

The public API in `include/openssl/` is considered stable within
major versions. Internal headers in `include/crypto/` and
`include/internal/` may change even between patch releases and are
not part of the public API. The provider interface is also
considered stable.

Components experiencing active development:

* The QUIC implementation is still evolving
* The provider architecture continues to mature

Long-standing stable components:

* Core cryptographic algorithms in `crypto/`
* The SSL/TLS protocol implementation (excluding QUIC)

File Organization Patterns
--------------------------

* `.c` files contain implementations
* `.h` files are headers
* `.pod` files are documentation sources (converted to man pages)
* `*_local.h` files are for internal use within a component
* `build.info` files describe what to build in each directory
* Test filenames typically include the name of what they test
